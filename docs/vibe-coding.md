# תהליך ה-Vibe Coding — איך נבנה MedTracker עם AI

מסמך זה מתעד את תהליך הבנייה בעזרת כלי AI (לבונוס המחוון).

## גישה כללית
המוצר נבנה בשלושה גלים, כשבכל גל ה-AI שימש גם *לכתיבת הקוד* וגם *בתוך המוצר*:
1. **אב-טיפוס מהיר** — אפליקציית HTML יחידה (RTL, נגישות לקשישים) לאימות הרעיון וה-UX.
2. **קשחת איכות** — תיקון באגים, נגישות (ARIA, יעדי מגע, ניגודיות), והפרדת קוד.
3. **מעבר ל-Production** — React + Supabase + Vercel לפי המחוון.

## איך השתמשנו ב-AI לאורך הבנייה
- **תכנון מודל הנתונים:** ניסוח הישויות (מטופל, תרופה, נטילה, מדד, יומן) והקשרים,
  ואז יצירת `schema.sql` עם enums, מפתחות זרים ומדיניות **RLS** מלאה.
- **ארכיטקטורה:** זיהוי שמעטפת `api` באב-הטיפוס היא "תפר" שמאפשר להחליף localStorage
  ב-Supabase בלי לשכתב את ה-UI — וכך תוכנן `lib/api.js`.
- **רכיבי React:** פירוק המסכים לרכיבים, חישובי עמידות כפונקציות טהורות (`lib/schedule.js`).
- **אבטחה:** העברת קריאות ה-AI ל-Edge Function (`claude-proxy`) כדי שמפתח Anthropic
  לעולם לא ייחשף בצד הלקוח.

## ה-AI בתוך המוצר
- **עוזר AI** — מענה על שאלות תרופות בהקשר הנתונים האישיים (תרופות, עמידות), עם
  הסתייגות רפואית והפניית חירום ל-101.
- **בדיקת אינטראקציות** — סריקת התרופות הפעילות וזיהוי אינטראקציות אפשריות לפי חומרה.
- **תובנה יומית** — הודעת עידוד מותאמת על בסיס עמידות ורצף.
כל אלה רצים דרך ה-Edge Function המאובטח.

## לקחים
- מעטפת נתונים אחת (`api`) הפכה את המעבר ל-Backend להדרגתי ובטוח.
- RLS מההתחלה — אבטחת מידע רפואי לא כתוספת בסוף.
- פונקציות טהורות לחישובי לוגיקה = קל לבדוק ולתחזק.
